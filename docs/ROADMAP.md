## 二、改进后采纳（与评审建议不同之处及理由）

- **存储升级（SQLite/DuckDB/向量库）**：采纳分层思路，但**零依赖核心是本项目
  的可复现承诺**，不作为默认依赖引入。方案：核心保持 JSONL+扁平索引（lite），
  规模化后端（FTS5/向量/图库）作为 `[scale]` optional-extra 增量实现，接口先行。
- **Poem → Work 十层数据模型**：不做一次性重写（会破坏全部下游与已审核规则）。
  渐进路径：Poem 已有 metrics/annotations 扩展位；下一版把 TextWitness（异文，
  已有雏形：also_in+互文重出）与 StructuralUnits（阕/联）作为独立层落盘，
  InterpretiveClaims 与 EvidenceGraph 由 ClaimGuard/审计日志演化而来。
- **五层诗义标注器**：字面层已有；句法/场景/诗学层需要 LLM 参与标注——采纳
  评审的「模型只出结构化候选、规则回源验证」设计，作为第二阶段 LLM 增益
  流水线实现（候选 JSON schema 已在评审建议中，直接沿用）。
- **平水韵/词林正韵/钦定词谱**：本轮已入《广韵》（更上游的权威源）；平水韵
  与词谱的可靠开放数据源仍在核查许可与版本，入库时同样保留来源层级。

## 三、四阶段路线（对齐评审建议）

**第一阶段（可信诗学基础）**：✅ ClaimGuard、QueryPlan、消歧、否定范围、
论证图字段、快慢测试、错误 schema、评测口径——本轮完成；
剩余：句法层标注器、专家论证的反证机制化。

**第二阶段（计算诗学）**：粘对/拗救完整化、对仗分析（词性/语义范畴）、
典故知识图谱（T5 互文）、结构仿拟检测（T4）、历代评点层、版本异文系统化
（TextWitness）、多路混合检索（语义向量作可选层）、CNPoetryBench
（含评审列出的七类对抗题）、事件溯源运行记录。

**第三阶段（现象级产品）**：「进入一首诗」诗境界面（分句节奏/视点推进/
情感曲线/可点意象）、用户画像与学习闭环（诊断→微课→练习→反馈）、
创作实验室（格律检查/意象建议/陈词识别，永远标注「今人拟作」）、
历代诗评议会（有出处的批评范式视角，禁止假扮古人发言）。

**第四阶段（全古典文学）**：ClassicalWork 统一接口，古文（句读/虚词/活用/
论证结构）、史传、辞赋、经史子集扩展——不把古文塞进 Poem 数据类。
